Interface Audit

With the Amoeba interface, I’ve found the biggest weakness is the lack of a beginning for action. Without any instructions for the tool itself, the website is left unexplained to the average viewer. While I think the size of the manipulated text does draw the user’s attention, which in turn could be considered as a cue to its interactivity, the lack of hints telling the viewing to click down to distort is a big issue in terms of properly communicating the purpose of the page to the user.

The main actions on the page consist of text manipulation, distortion manipulation, and type distortion. The large text is clickable and interactive but unclear which is the biggest issue of the interface.

The page is fairly condensed as it only exists on the dimensions of the front. This extends out past the webpage size but only for the js canvas so to allow for more room for type to be manipulated.

The biggest assumptions of the page are mainly that of the user understanding how the page and tools work/are enacted. This is a big flaw. One other small consideration concerning constraints is the constraints on retaining or saving certain creations made through the distortion. This is something which I hope to potentially implement since it allows for people to more efficiently organize and utilize any creations made through Amoeba.

I think one heuristic which my page violates is the match between the system and the real world. As I have mentioned before, there is a noticeable lack of instructions and/or explanations on my page. This lack of visual communication creates a distinct disconnect between my system and the way actions are often facilitated by verbal or visual cues in the real world. This will be fixed by additional imagery/instructions. Additionally, I think another area where my page is particularly lacking is in Error Prevention. I’ve had it occur multiple times where I accidentally clicked reset or revert which led to me losing the visual I wanted or was happy with. This leads me to another addition I would like to incorporate which is an undo/ go back button. It would be especially fun if that function could happen fluidly and you could watch the change to the type which has occurred in reverse (and potentially sped up or slowed down).

My three structural changes are this: Enlarge the text box so that it is more visible and add instructions nearby to facilitate the users properly using the page Add demo imagery/instructions for manipulating type. This can either be in the form of an infographic on how to use the tool or just words telling the user to press down on the letters. Create some function which will allow the user to see/revert back to previous states in the type manipulation. This will give better state visibility as well as work towards error prevention.
